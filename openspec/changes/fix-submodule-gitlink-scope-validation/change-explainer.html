fix-submodule-gitlink-scope-validation

修正 goal-runner controller scope validation：當父 repo 只呈現 Git submodule gitlink（例如 aos-core）變更時，必須檢查 submodule 內部 diff 是否真的落在節點 allowedPaths / forbiddenPaths 範圍內。這不是放寬 scope，而是把目前看不到的內部 diff 還原成可驗證的 workspace-relative paths。

Value path: proceed_to_specNo public schema change expectedFail closedSource of truth: proposal/design/tasks/spec

Why 為什麼需要這個變更

Goal 6fc8f3ab 顯示，合法的 submodule-scoped 工作會因父 repo 只呈現 aos-core gitlink 而被 controller validation 阻擋。這讓已完成的節點無法成為可整合結果，依賴節點也無法前進。

What Changes 變更內容

  • 辨識 relevant submodule gitlink。
  • 解析 old/new gitlink revisions。
  • 取得 internal diff 並映射回 workspace-relative paths。
  • 套用既有 allowedPaths / forbiddenPaths。
  • 缺少證據時 fail closed。

Affected Surfaces 影響面

主要影響 goal-runner controller validation 與 focused tests；不預期改 public schema、持久化資料、monitor UI 或 submodule publish policy。

Scope Boundaries 範圍邊界

此 change 只處理 path-policy validation。它不把整個 submodule root 當成 allowed path，不要求 graph authors 放寬 scope，也不修復歷史 blocked goal。

Rollout 推出與回滾

這是內部 validation behavior change；以 regression tests 和 standard repository check 驗證後推出。回滾方式是 revert 相關 code change，無資料 migration。

Current Status 目前狀態

OpenSpec proposal/design/tasks/spec 已建立。本頁是 companion decision-review view，不取代 markdown/spec source of truth。

Before / After 對照

變更前 before

完成的 submodule-scoped 節點在父 repo 中只呈現 aos-core gitlink 變更。controller 直接把 aos-core 與 aos-core/packages/runtime-ports/** 比對，判定 outside allowed paths，導致節點與依賴節點 blocked。

變更後 after

controller 先辨識 aos-core 是 relevant submodule gitlink，解析 old/new revisions，取得 internal diff，映射成 aos-core/packages/... paths，再套用既有 allowed/forbidden policy。只有全部通過才接受 gitlink。

Decision Points 決策點

選擇：submodule-aware validation

在 controller validation 層理解 gitlink，並用 internal diff 驗證。這保留嚴格 scope，又讓合法 submodule 節點可通過。

Recommendation: 採用

拒絕：把 aos-core 加進 allowedPaths

這是短期 workaround，但會把整個 submodule root 放行，無法證明 gitlink 裡沒有 unrelated 或 forbidden path。

Decision: 不採用

保留為 backlog：integration boundary cleanup

未來可整理父 repo gitlink 更新的責任邊界，但仍不能取代 validation 必須理解最終 gitlink 代表的內部變更。

Decision: 延後

Implementation Slices 實作切片

Slice 1 · Gitlink classification

辨識 changed path 是否為 relevant Git submodule root，並確認 policy 中存在該 root 下的 scoped paths。

Acceptance: 普通非 submodule path 行為不變。

Slice 2 · Internal diff mapping

解析 old/new gitlink revisions，取得 submodule internal changed paths，映射回 workspace-relative paths。

Acceptance: 無法解析時 fail closed。

Slice 3 · Policy integration

把 mapped paths 送進既有 allowedPaths / forbiddenPaths matching，產生 pass 或明確 policy failure。

Acceptance: allowed/disallowed/forbidden 測試全部覆蓋。

Verification Plan 驗證計畫

  • 測試 aos-core gitlink 內部 diff 只包含 aos-core/packages/runtime-ports/** 時通過。
  • 測試 internal diff 包含 aos-core/packages/domain-adapters/** 時，在只允許 runtime-ports 的節點中失敗。
  • 測試 mapped path 命中 forbiddenPaths 時失敗。
  • 測試 old/new revision 或 diff 不可得時 fail closed 且 diagnostic 清楚。
  • 保留 ordinary path outside allowed 的 regression test。

Risk Register 風險清單

高風險：錯誤放行整個 submodule root

Mitigation: 永遠檢查 internal diff；root path 本身不得因 child allowedPath 而直接 pass。

中風險：local revision 不存在

Mitigation: fail closed，不做隱性 pass；diagnostic 說明缺少哪個 revision。

中風險：影響普通 path validation

Mitigation: 加非 submodule regression test，保持既有 matchesPathPolicy 行為。

低風險：診斷訊息不夠可操作

Mitigation: 訊息包含 submodule root 與不能驗證的原因。
